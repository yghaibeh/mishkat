## ١. لنبدأ من عالمك: خطأ لن تراه إلا متأخراً

اكتب معي هذه الدالة في Python:

```python
def add_payment(amount, currency):
    return {"amount": amount, "currency": currency}

# في مكان بعيد من المشروع، بعد ثلاثة أشهر:
add_payment("100", "USD")   # ⬅️ مبلغ نصّي "100" لا رقم 100
```

Python **لن تعترض**. الدالة ستعمل، والقيمة ستُخزَّن، وكل شيء يبدو سليماً... حتى يأتي سطر بعيد يحسب `total = amount * 12` فينتج `"100100100100..."` (تكرار النص ١٢ مرة!) أو ينفجر بـ`TypeError` — **في production، أمام المستخدم، الساعة ٢ فجراً**.

هذا هو جوهر المشكلة: في Python الخطأ موجود منذ لحظة كتابته، لكنه **مختبئ** — لا يظهر إلا حين يمرّ التنفيذ فوق ذلك السطر بالذات، بتلك القيمة بالذات.

الآن نفس الدالة في TypeScript:

```typescript
function addPayment(amount: number, currency: string) {
  return { amount, currency }
}

addPayment("100", "USD")
//         ^^^^^ ❌ خطأ فوراً — قبل أن يعمل البرنامج أصلاً:
// Argument of type 'string' is not assignable to parameter of type 'number'
```

الخطأ **نفسه**، لكنه ظهر **لحظة الكتابة** لا بعد ثلاثة أشهر. هذا كل شيء. هذه هي الفكرة كلها. كل ما تبقى تفاصيل على هذه الجملة:

> **الـtype system ينقل اكتشاف صنف كامل من الأخطاء من «وقت التشغيل عند المستخدم» إلى «وقت الكتابة عندك».**

## ٢. المصطلحات الأساسية — بهدوء

### Static vs Dynamic typing

- **Python = dynamically typed**: نوع القيمة يُعرف **وقت التشغيل (runtime)**. المتغير `x` قد يكون رقماً الآن ونصاً بعد سطرين، ولا أحد يعترض.
- **TypeScript = statically typed**: نوع كل متغير ودالة معروف **قبل التشغيل**، وهناك برنامج اسمه **compiler** (المُترجم — عندنا اسمه `tsc` أي TypeScript Compiler) يقرأ الكود كله ويتحقق أن الأنواع متوافقة، **قبل** أن يُسمح للكود بالعمل.

فكّر في `tsc` كمدقق يقرأ مشروعك كاملاً في ثوانٍ ويقول: «في الملف الفلاني سطر كذا، أنت تمرر string لدالة تنتظر number». مجاناً. في كل مرة.

### Compile time vs Runtime

- **Compile time**: لحظة فحص/تحويل الكود — قبل أن يعمل.
- **Runtime**: لحظة تنفيذ الكود فعلاً — عند المستخدم أو في الاختبار.

القاعدة الذهبية التي ستسمعها مني كثيراً في هذا الكورس:

> **كل خطأ تمسكه في compile time هو خطأ لن تحتاج اختباراً له، ولن يصل للميدان أبداً.**

### وماذا عن type hints في Python؟

سؤال ممتاز ستسأله حتماً: «لكن Python فيها type hints!»

```python
def add_payment(amount: int, currency: str) -> dict: ...
```

صحيح — والفرق **جوهري**:

| | Python type hints + mypy | TypeScript |
|---|---|---|
| هل الفحص إلزامي؟ | **اختياري** — الكود يعمل حتى لو تجاهلت كل الـhints | **إلزامي** — لا build مع خطأ types |
| من يفرضه؟ | أداة خارجية (mypy) إن شغّلتها | الـcompiler نفسه، دائماً |
| الثقافة | أغلب مشاريع Python لا تفرضه بصرامة | لا يوجد TS بدون فحص |

في مشروعنا، هذا الفرض له اسم: **البوابة G1**. أول بوابة من بواباتنا الـ٢٢ هي حرفياً «شغّل `tsc` — إن وُجد خطأ نوعٍ واحد، الدمج مرفوض». أي أن **كود مشكاة لا يمكن فيزيائياً أن يصل إلى main وفيه خطأ types**. هذا ليس التزاماً أخلاقياً من المبرمج — إنه حارس آلي.

## ٣. Duck typing و Structural typing — ستحب هذه

في Python تعرف مبدأ **duck typing**: «إن مشى كالبطة وصاح كالبطة فهو بطة» — لا يهمني class الكائن، يهمني أن عنده الـmethods التي أحتاجها.

خبر جيد: TypeScript تعمل **بنفس الروح**، لكن في compile time. اسمها **structural typing** — التوافق يُقاس **بشكل (structure) الكائن** لا باسمه:

```typescript
type Point = { x: number; y: number }

function distance(p: Point) { /* ... */ }

const myThing = { x: 3, y: 4, color: "red" }
distance(myThing)   // ✅ يمرّ! myThing «شكله» يطابق Point (فيه x وy رقميان)
                    // الحقل الزائد color لا يضر
```

لاحظ: `myThing` لم يُعلَن أنه `Point`. لا وراثة ولا تسجيل. **الشكل يكفي** — تماماً كـduck typing، لكن الفحص يتم قبل التشغيل لا أثناءه. الانتقال الذهني من Python إلى TS أسهل مما تظن لهذا السبب بالذات: الفلسفة نفسها، التوقيت مختلف.

## ٤. قصة الـ«لا شيء»: من None إلى null و undefined

في Python عندك «لا شيء» واحد: `None`. في JavaScript/TypeScript عندك **اثنان**، وهذا يربك كل قادم من Python:

- **`null`** = «لا قيمة، **عن قصد**» — أحدهم قرر صراحة أن هذا فارغ.
- **`undefined`** = «لم يُعطَ قيمة أصلاً» — حقل لم يُملأ، متغير لم يُسند، parameter لم يُمرَّر.

لماذا يهمك الفرق؟ لأنه **ليس فرقاً أكاديمياً — إنه فرق بين معنيين تجاريين مختلفين**. وعندنا في المشروع قصة حقيقية وقعت قبل أيام تشرحها أفضل من أي مثال مصطنع:

> **قصة من كودنا (CR-028)**: سجل التدقيق (audit log) عندنا يحفظ مع كل عملية حساسة لقطتين: `before` (كيف كان الشيء قبل التعديل) و`after` (كيف صار). في البداية كان الحقلان **اختياريين** (`before?: string`). ثم اكتشفنا المشكلة: حين يصل قيد تدقيق بلا `before`، ماذا يعني؟
> - هل يعني «هذه عملية إنشاء، لا يوجد before أصلاً» (فراغ **مشروع**)؟
> - أم يعني «المبرمج **نسي** تمرير before» (فراغ **خطأ**)؟
>
> **لا أحد يستطيع التفريق.** الحالتان تبدوان متطابقتين في البيانات، وعلاجهما متعاكس. فقرارنا كان: الحقلان **إلزاميان**، ومن لا لقطة عنده يكتب `null` **صراحة** — أي يقول بصوت عالٍ: «أنا واعٍ أن لا لقطة هنا». الآن الفراغ المقصود `null` والنسيان **لا يُترجم أصلاً** — `tsc` يرفضه.

احفظ هذه الجملة، ستتكرر في الكورس كثيراً:

> **الغياب الصامت عدوّ. النظام الجيد يجبرك أن تقول الغياب بصوت عالٍ.**

## ٥. الآن نفتح كودنا الحقيقي — أول ثلاثة توقيعات

### التوقيع الأول: أبسط من Python نفسها

من `v2/src/features/ledger/data/store.ts` (دفتر المحاسبة):

```typescript
fundBalance(fundId: string, currency: string): Cents
```

اقرأها هكذا: «دالة اسمها fundBalance، تأخذ `fundId` نصاً و`currency` نصاً، **وتعيد `Cents`**». في Python كنت ستكتب docstring يشرح هذا وتتمنى ألا يكذب مع الزمن. هنا التوقيع **هو** التوثيق — ولا يستطيع أن يكذب، لأن الـcompiler يفرضه على كل مستدعٍ وكل سطر داخل الدالة.

### لكن مهلاً — ما هو `Cents`؟ (أول لقاء مع فكرة عميقة)

`Cents` ليس نوعاً موجوداً في اللغة. **نحن صنعناه**. لماذا لم نكتفِ بـ`number`؟

لأن `number` يسمح بهذه الكارثة الصامتة:

```typescript
const priceInDollars = 10.50       // دولارات
const balanceInCents = 45000       // سنتات
const total = priceInDollars + balanceInCents   // 45010.5 ⬅️ هراء محاسبي، ولا أحد اعترض
```

كلاهما `number`، فالجمع «سليم» في نظر اللغة — و**فاسد** في نظر المحاسبة. عملة النظام عندنا كلها بالسنت (integer) تحديداً لتفادي كسور الفاصلة العائمة، فصنعنا نوعاً خاصاً (يسمى هذا النمط **branded type** — سنفصّله في درس لاحق، اليوم يكفيك المبدأ):

> إن كان لرقمك **معنى** (سنتات، دقائق، نقاط)، أعطه **اسماً في الـtype system** — فيصير خلطُ سنتاتٍ بدولارات أو دقائقَ بساعات **خطأَ ترجمة** لا خطأ إنتاج.

هذا خط دفاع لا وجود له في Python العادية إطلاقاً: **الخطأ المنطقي تحوّل إلى خطأ نوعٍ يُمسك قبل التشغيل**.

### التوقيع الثالث: نوع يحكي قصة

من `v2/src/db/unitOfWork.ts` (طبقة قاعدة البيانات):

```typescript
export type Scope = {
  readonly tenantId: string
  /** بادئةُ المسار المحمَّلة — `/` تعني الشبكةَ كلَّها. */
  readonly scopePath: string
}
```

ثلاث ملاحظات، كل واحدة درس مصغّر:

1. **`type Scope = {...}`** — تعريف «شكل» (مثل dataclass في Python لكن بلا سلوك، شكل فقط).
2. **`readonly`** — هذا الحقل **لا يمكن تعديله بعد الإنشاء**. جرّب `scope.tenantId = "x"` في أي مكان ⟵ خطأ ترجمة. (لماذا نُجمّد كل شيء؟ درس ٠-٤ كامل عن هذا).
3. الأهم — **ما ليس موجوداً**: لاحظ أن `Scope` فيه **محور واحد** للتضييق (المسار). حين احتجنا نناقش «هل نضيف تضييقاً بالشخص؟» كان النقاش كله يدور حول **هذا التعريف تحديداً** — هل نضيف حقلاً هنا؟ الـtype صار **وثيقة القرار المعماري نفسها**. في الشركات الكبرى، مراجعة الـtypes هي ٨٠٪ من مراجعة التصميم — لأن الأنواع تكشف البنية أصدق من أي رسم.

## ٦. لماذا تعتبره الشركات العالمية خط الدفاع الأول؟

أربعة أسباب عملية، لا نظرية:

**١) الـRefactoring بلا رعب.** أعدنا تسمية حقل `scopePath` إلى `unitPath` في سجل التدقيق قبل أيام — عبر ملفات كثيرة. في Python: بحث نصي ودعاء ألا تفوتك حالة أو تصيب حالة مشابهة بالاسم فقط. في TS: غيّر التعريف، شغّل `tsc`، **فيسرد لك كل موضع يجب تعديله بالملف والسطر** — وحين يصمت، انتهيت فعلاً. المشاريع الضخمة (Google/Microsoft) لا تستطيع العيش بدون هذا؛ لهذا كتبت Microsoft لغة TS أصلاً.

**٢) التوقيع عقد لا يكذب.** الـdocstring يشيخ ويكذب؛ التوقيع مفروض بالقوة. حين تقرأ `fundBalance(fundId: string, currency: string): Cents` فأنت تعرف **يقيناً** ما يدخل وما يخرج.

**٣) أصناف كاملة من الأخطاء تنقرض.** «مرر string مكان number»، «نسي حقلاً في الكائن»، «استدعى method غير موجودة»، «تعامل مع null كأنه قيمة» — هذه تشكل نسبة معتبرة من أخطاء production في اللغات الديناميكية، وفي TS الصارم **لا تصل إلى git أصلاً**.

**٤) الحارس الذي لا يمكن نسيانه.** أفضل مثال حدث عندنا قبل يومين: حين دققتُ إلزام `before/after` في سجل التدقيق، حاولت كسره — أعدت الحقل اختيارياً كما كان. النتيجة؟ **الكود لم يُترجم أصلاً**. قارن: لو كان الحارس اختباراً، يستطيع مهمل حذفه أو تجاهله؛ أما حارس مبني في الـtype فلا سبيل حوله إطلاقاً. القاعدة عندنا:

> **أقوى حارس هو الذي يجعل الخطأ غير قابل للكتابة، لا الذي يمسكه بعد كتابته.**

(الدرس ٠-٣ سيريك ذروة هذا المبدأ: discriminated unions — كيف جعلنا «راتباً صفراً بلا سبب» حالة **يستحيل تمثيلها** في الذاكرة.)

## ٧. الأمانة العلمية: ماذا **لا** يمسكه الـtype system؟

حتى لا تخرج بفكرة أن types = أمان كامل. الـcompiler يفحص **الأشكال، لا المعاني**:

```typescript
function transfer(from: string, to: string, amount: Cents) { ... }

transfer(toAccount, fromAccount, amount)
//       ^^^^^^^^^  ^^^^^^^^^^^ ⬅️ معكوسان! وكلاهما string فالأنواع «سليمة» ✅
```

خطأ منطقي كارثي، والأنواع راضية تماماً. النوع يضمن «رقم حيث يجب رقم»، لا «الرقم الصحيح». ولهذا:

- **الأنواع** تمسك أخطاء **الشكل** ⟵ مجاناً وفوراً.
- **الاختبارات** تمسك أخطاء **المعنى** ⟵ وهي المرحلة ٢ كاملة من كورسنا.
- وعندنا **٤٥٠٠+ اختبار فوق** الأنواع الصارمة — طبقتان لا تُغني إحداهما عن الأخرى.

## ٨. تمرين الدرس (١٠ دقائق — بلا حاسوب)

اقرأ التواقيع وأجب، ثم قارن بالحلول آخر الملف:

**س١**: من كود الرواتب عندنا:
```typescript
function hoursTrack(ctx: PayrollContext, input: DeriveInput, personId: string): TrackOutcome
```
أ) كم parameter تأخذ؟ ب) ماذا تعيد؟ ج) هل تستطيع من التوقيع وحده معرفة إن كانت تقرأ قاعدة البيانات؟

**س٢**: لماذا **يرفض** الـcompiler هذا؟
```typescript
type Currency = "USD" | "TRY" | "SYP"
const c: Currency = "EUR"
```
(تلميح: هذا union من **literal types** — النوع هنا ليس «أي نص» بل «أحد هذه النصوص الثلاثة حصراً».)

**س٣**: في Python كتبت `def f(x=None)` واستعملت `if x is None` للتفريق بين «لم يُمرَّر» و«مُرّر». ما مشكلة هذا النمط التي حلّها قرارنا في CR-028؟ (بكلماتك)

**س٤ (للتشغيل إن أردت)**: افتح terminal في `v2/` واكتب `npx tsc --noEmit`. ماذا تتوقع أن ترى، ولماذا هذا بالضبط ما تريد رؤيته؟

## ٩. كيف تحكم بهذا الدرس على كودٍ أو تسليم AI؟

من اليوم، حين تنظر في أي كود TS (من وكيل أو إنسان)، اسأل:

- ☑️ هل أرى `any` في التواقيع؟ (`any` = «أطفئ الفحص هنا» — عندنا **بوابة G3 تحظره** حظراً تاماً. وجوده في تسليم = علامة حمراء فورية.)
- ☑️ هل الأرقام ذات المعنى لها أنواع باسمها (`Cents`) أم `number` عائم في كل مكان؟
- ☑️ هل الغياب صريح (`null` معلن) أم صامت (حقول اختيارية `?` تُترك للتخمين)؟
- ☑️ هل أرى `as` كثيراً؟ (`as` = «صدّقني بلا فحص» — قليله ضرورة، كثيره تهريب أخطاء من تحت أنف الـcompiler.)

## الحلول

**ح١**: أ) ثلاثة: السياق، مدخلات الاشتقاق، ومعرّف الشخص. ب) `TrackOutcome` — وهو union إما «حصيلة» وإما «سبب صمت» (تفصيله درس ٠-٣). ج) **لا تعرف من التوقيع** — لكن الملاحظة الذكية: الدالة تأخذ `ctx` جاهزاً، وفي عمارتنا الخدمات لا تستعلم بنفسها بل تُعطى كل شيء عبر السياق (درس ٣-٤). التوقيع يلمّح، والعمارة تؤكد.

**ح٢**: لأن `Currency` معرَّف كقائمة مغلقة من ثلاث قيم حرفية، و`"EUR"` ليست منها. لاحظ القوة: «العملات المدعومة» صارت **قاعدة يفرضها الـcompiler** — في Python كانت ستكون توثيقاً أو فحص runtime قد يُنسى.

**ح٣**: `None` الواحد لا يفرّق بين «غياب مقصود» و«نسيان». الحل: تصميم الـAPI بحيث يكون الغياب **تصريحاً إلزامياً** (`null` صريح والحقل إلزامي) — فيستحيل النسيان لأنه لا يُترجم.

**ح٤**: **لا شيء** — يصمت وينتهي بنجاح. صفر أخطاء types في المشروع كله، وهذا ليس اجتهاداً بل **بوابة G1**: أي خطأ types = لا دمج. الصمت هنا هو الصوت الأجمل.
