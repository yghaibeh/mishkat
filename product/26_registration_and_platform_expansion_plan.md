## الجزء الأوّل — (٣) التسجيل الذاتيّ مع اعتمادٍ هرميّ

### ١. الفكرة والمبدأ الحاكم

بدل أن تُضيف الإدارةُ كلَّ شخصٍ يدويًّا («الإضافة الذاتية»)، يسجّل الشخصُ **نفسَه** من صفحةٍ عامّة، ويعتمده **صاحبُ الطبقة المغطّية فوقه مباشرةً**:

| المسجِّل | مَن يعتمده |
|---|---|
| طالبٌ في مسجد الفاروق | مدير مسجد الفاروق (الأمير) |
| معلّم/مسؤول حلقة في مسجد | مدير المسجد |
| مسؤول مسجد (حتى لو المسجد **غير مُنشأ** بعد) | مسؤول المربع، وإلّا مسؤول المنطقة، وإلّا المدير العام |
| مسؤول مربع | مسؤول المنطقة، وإلّا المدير العام |
| مسؤول منطقة | المدير العام |

**القاعدة الذهبية (من رسالة العميل):** «المسجد يُضاف حين الاعتماد» — لو سجّل شخصٌ نفسَه مديرًا لمسجد الأمة وهو غير موجودٍ عندنا، فعند الاعتماد يُنشئ البرنامجُ **كلَّ الهيكليّة** ذرّيًّا: الوحدة التنظيمية (المسجد تحت المربع/المنطقة المختارة) + الشخص + الحساب + التكليف المعتمَد.

### ٢. القرارات المعماريّة (بعد تفكيرٍ وتكرار)

1. **لا حساب قبل الاعتماد.** الطلب يعيش في جدول `registration_requests` مكتفيًا بذاته. لماذا؟
   - أمان: لا حسابات نصفيّة يمكن الدخول بها أو تعداد أسمائها.
   - نظافة: الرفض لا يترك أثرًا في `users/persons`.
   - التكامل مع دورة حياة الحساب القائمة يبقى بسيطًا (لا حالة `pending` جديدة في users).
2. **كلمة المرور تُخزَّن مُجزّأةً فورًا** بـ`hashPassword` (pbkdf2) القائم — لا نصّ صريح أبدًا، حتى في الطلب المعلّق.
3. **متابعة الطلب برمز (token):** بعد التقديم يُعطى المسجِّل رمزَ متابعةٍ (uuid) وصفحة `/register?status=<token>` تعرض: قيد المراجعة / مقبول (يمكنك الدخول) / مرفوض + السبب. لا يحتاج حسابًا للاستعلام.
4. **توجيه الاعتماد ديناميكيّ لا مُخزَّن:** صندوق «طلبات الانضمام» يُحسَب عند القراءة (كصندوق «بانتظار اعتمادك») — يظهر الطلبُ لكلّ مَن **يغطّي** مسارَ الهدف **وطبقتُه أعلى من الدور المطلوب**؛ فإن تغيّر المشرفون بين التقديم والاعتماد بقي التوجيه صحيحًا.
5. **حجز اسم الدخول:** فريد عبر `users.login` **و** الطلبات المعلّقة معًا (تحقّق عند التقديم وعند الاعتماد — الاعتماد هو الحكم الأخير لأنّ التسابق ممكن).
6. **الاعتماد ذرّيّ (batch):** إنشاء (المسجد إن لُزم) + person + person_contacts + user + role_assignment + إشعارات + audit في `db.batch` واحد — إمّا كلّه أو لا شيء.
7. **مكافحة إساءة بسيطة وفعّالة:** حقل honeypot خفيّ + سقف طلباتٍ معلّقةٍ لنفس اسم الدخول/الهاتف + أطوال مضبوطة بـzod. (لا CAPTCHA خارجيّ — يمكن لاحقًا Cloudflare Turnstile.)
8. **الطالب يصبح هويّةً كاملة:** عند اعتماد طالبٍ نُنشئ person+user (ليستفيد لاحقًا من وحدات «المطلوب اليوم/الاختبارات») ونضيف صفَّ `circle_students` مربوطًا بعمود `person_id` الجديد (هجرة إضافيّة عمودًا واحدًا) إن اختار حلقةً — فتلتقي هويّةُ الدخول بسجلّ الحلقة.

### ٣. نموذج البيانات — هجرة `0040_registration_requests`

```sql
CREATE TABLE registration_requests (
  id TEXT PRIMARY KEY,                 -- uuid = رمز المتابعة العام
  kind TEXT NOT NULL,                  -- student | teacher | amir | square | rabita
  full_name TEXT NOT NULL,
  gender TEXT NOT NULL DEFAULT 'male',
  login TEXT NOT NULL,
  password_hash TEXT NOT NULL,         -- pbkdf2 فورًا
  phone TEXT,
  -- الهدف في الهيكل:
  target_unit_id TEXT,                 -- الوحدة القائمة (مسجد/مربع/منطقة) إن وُجدت
  target_path TEXT,                    -- مسارها (منسوخ للاستعلام السريع)
  proposed_unit_name TEXT,             -- اسم مسجدٍ جديدٍ مقترح (حالة «مسجدي غير موجود»)
  proposed_parent_id TEXT,             -- الأب الذي سيُنشأ تحته المسجد الجديد (مربع/منطقة)
  circle_id TEXT,                      -- للطالب: الحلقة المختارة (اختياري)
  note TEXT,                           -- تعريف مختصر بالنفس (اختياري)
  status TEXT NOT NULL DEFAULT 'pending',  -- pending | approved | rejected
  decided_by TEXT, decided_by_name TEXT, decided_at INTEGER,
  reject_reason TEXT,
  created_unit_id TEXT,                -- المسجد الذي أنشأه الاعتماد (للتدقيق)
  created_person_id TEXT, created_user_id TEXT,
  created_at INTEGER NOT NULL
);
CREATE INDEX idx_regreq_status ON registration_requests(status);
CREATE INDEX idx_regreq_login ON registration_requests(login);
CREATE INDEX idx_regreq_target ON registration_requests(target_path);

ALTER TABLE circle_students ADD COLUMN person_id TEXT; -- ربط هوية الطالب المعتمَد بسجل الحلقة
```

**ثابت الأدوار المسموح تسجيلُها ذاتيًّا:** `student | teacher | amir | square | rabita` (لا admin ولا section_head ذاتيًّا — حسّاسان، يبقيان بالتعيين).

### ٤. خوارزميّة توجيه الاعتماد (تُنفَّذ في القراءة والاعتماد)

```
level(kind): student/teacher→4(داخل المسجد) · amir→3 · square→2 · rabita→1
target_path(req):
  target_unit_id موجود ⇒ مسار الوحدة
  اقتراح مسجد جديد ⇒ مسار proposed_parent (فالمعتمِد مَن يغطّي الأب)
canApprove(user, req):
  admin ⇒ نعم دائمًا
  student/teacher ⇒ أمير ذلك المسجد تحديدًا (org_unit_id = target)
                    أو أي طبقة تغطيه (square/rabita/section_head)
  amir ⇒ square/rabita/section_head يغطّي target_path
  square ⇒ rabita/section_head يغطّي
  rabita ⇒ section_head يغطّي أو admin
```
الأولويّة في الإشعار: **أقرب طبقةٍ مغطّية** (كما في تذكيرات الإشراف الميدانيّ)؛ لكن الصندوق يظهر لكلّ مخوَّل حتى لا يعلق طلبٌ إذا غاب الأقرب.

### ٥. مسارات الخادم (server functions)

| الدالة | عامّة؟ | الوصف |
|---|---|---|
| `publicOrgTreeData()` | ✅ بلا جلسة | شجرة مختصرة: الأقسام → المناطق → المربعات → المساجد (أسماء+ids النشطة فقط) + حلقات كل مسجد للطالب. لا أسماء أشخاص، لا إحصاءات — خصوصية. |
| `submitRegistrationData(input)` | ✅ بلا جلسة | zod صارم، honeypot، فحص فريد login (users+pending)، سقف تكرار، hash، إدراج pending، إشعار `registration_pending` للمخوَّلين، إرجاع token. |
| `registrationStatusData(token)` | ✅ بلا جلسة | pending/approved/rejected + السبب — بالرمز فقط. |
| `pendingRegistrationsData()` | 🔒 | طلبات canApprove(user) ⇒ للصندوق. |
| `approveRegistrationData(id)` | 🔒 | تحقّق الصلاحية ثمّ **ذرّيًّا**: (مسجد جديد إن لُزم بوراثة section/genderTrack من الأب) + person + contacts + user + role_assignment approved (+circle_students للطالب مع person_id) + audit + تحديث الطلب. إعادة فحص فريد login هنا أيضًا. |
| `rejectRegistrationData(id, reason)` | 🔒 | سبب إلزامي، تحديث الحالة + audit. |

**تكامل الإشعارات:** نوع جديد `registration_pending` (نصّ: «طلب انضمام جديد: فلان — مسؤول مسجد "الأمة"») في الجرس + تيليغرام + Push، بنفس نمط `notifText`/`buildMessage`.

### ٦. الواجهات

1. **`/register` (عامّة):** معالج من ٣ خطوات RTL بنمط التطبيق:
   - **الدور:** بطاقات (طالب / معلّم حلقة / مسؤول مسجد / مسؤول مربع / مسؤول منطقة).
   - **الموقع:** قوائم متسلسلة (قسم → منطقة → مربع → مسجد → حلقة حسب الدور). لمسؤول المسجد: مفتاح «مسجدي غير مدرج» ⇒ حقل اسم المسجد + اختيار المربع/المنطقة الأب.
   - **البيانات:** الاسم الكامل، اسم الدخول (فحص توفّرٍ لحظيّ)، كلمة المرور (قياس قوّة بسيط)، الهاتف، تعريف مختصر. + honeypot خفيّ.
   - **النجاح:** بطاقة «قُدِّم طلبك» + رمز المتابعة + زرّ نسخ + رابط الاستعلام.
2. **رابط من `/login`:** «ليس لديك حساب؟ سجّل الآن».
3. **صندوق «طلبات الانضمام»** في لوحة الشبكة (بجانب «بانتظار اعتمادك») يظهر لمن لديه طلبات: بطاقة لكلّ طلب (الاسم، الدور، الموقع/«جديد: مسجد الأمة تحت مربع كذا»، الهاتف، النبذة) وزرّا **قبول** (تأكيد) و**رفض** (سبب إلزامي).
4. **صفحة الاستعلام:** `/register` مع `?status=<token>`.

### ٧. الأمن والحواف (مدروسة واحدةً واحدة)

- تعداد أسماء الدخول: فحص التوفّر يُرجع متوفّر/غير متوفّر فقط (لا يكشف لمن).
- سباق login متزامن: الفحص يعاد داخل الاعتماد؛ فشلُه يعيد خطأً واضحًا للمعتمِد («اسم الدخول حُجز، اطلب من المتقدّم اسمًا آخر» — زر رفض بسبب جاهز).
- مسجد مقترح مكرّر الاسم تحت نفس الأب: تحذير للمعتمِد + خيار «اربط بالمسجد الموجود بدل الإنشاء».
- وحدة هدف حُذفت/جُمّدت قبل الاعتماد: يظهر الطلب بشارة «الوحدة لم تعد نشطة» والقبول محجوب.
- التقديم لا يفشي وجود المستخدم: رسائل عامة.
- كلّ قرارات القبول/الرفض في `audit_log`.

### ٨. الاختبارات (vitest)

1. تقديم طالبٍ لمسجدٍ ⇒ pending ويظهر لأمير المسجد فقط دون أمير مسجدٍ آخر.
2. اعتماد أمير-مسجدٍ **جديدٍ** ⇒ تُنشأ الوحدة تحت المربع بمسارٍ صحيح + person+user+تكليف approved، والدخول يعمل (hash صالح).
3. مسؤول مربع يعتمده مسؤول المنطقة ولا يعتمده أمير مسجد.
4. رفضٌ بسبب ⇒ status=rejected والاستعلام بالرمز يعيد السبب.
5. login مكرّر (قائم أو معلّق) ⇒ رفض التقديم؛ وسباقُه عند الاعتماد يفشل بأمان.
6. الطالب مع حلقة ⇒ صفّ circle_students موصول بperson_id.
7. الإشعارات: التقديم يُنشئ `registration_pending` للطبقة الصحيحة، بلا تكرار.

## الجزء الثاني — (٤) وحدات التوسعة المستوحاة من «مجلس»

> ✦ **أولويّة العميل المنصوصة:** «الملفّات التدريبيّة — عشرات الملفات، يتابع البرنامجُ استلامَ كلّ مسؤول مسجدٍ لها وإنجازَ قراءتها ضمن نظامٍ جميل» ⇒ **تُنفَّذ أوّلًا (ت١-ت٣)**، ثم بقية الوحدات.

### ت) مكتبة الموادّ والملفّات التدريبيّة + متابعة الإنجاز ✦

**الفكرة:** مكتبة موادّ مصنّفة (عقيدة/فقه/سيرة/تربية/إداريّ-تدريبيّ…) بملفّات PDF/صوت في R2، تُسنَد إلى فئةٍ مستهدفة (مسؤولو المساجد، المعلّمون…)، ويتتبّع النظام لكلّ فردٍ: **استلم → فتح → أنجز (إقرار قراءة)**، مع لوحة متابعةٍ للإدارة وتذكيراتٍ آليّة.

**هجرة `0041_training_library`:**
```sql
CREATE TABLE materials (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  category TEXT NOT NULL,              -- aqeedah|fiqh|seerah|tarbiya|admin_training|other
  kind TEXT NOT NULL,                  -- pdf | audio | link
  r2_key TEXT, external_url TEXT, content_type TEXT, size_bytes INTEGER,
  description TEXT,
  audience TEXT NOT NULL DEFAULT 'amir',   -- amir|teacher|supervisor|all — الفئة المستهدفة
  mandatory INTEGER NOT NULL DEFAULT 0,    -- إلزاميّ ⇒ يدخل في المتابعة والتذكير
  sort_order INTEGER NOT NULL DEFAULT 0,
  status TEXT NOT NULL DEFAULT 'active',   -- active|archived
  created_by TEXT, created_at INTEGER NOT NULL
);
CREATE TABLE material_progress (
  id TEXT PRIMARY KEY,
  material_id TEXT NOT NULL,
  person_id TEXT NOT NULL,
  delivered_at INTEGER,                -- أُتيح له (أوّل ظهور في مكتبته)
  opened_at INTEGER,                   -- أوّل فتح فعليّ
  completed_at INTEGER,                -- أقرّ بالإنجاز
  UNIQUE(material_id, person_id)
);
```
**قرارات:**
- «الاستلام» يُختم آليًّا أوّلَ مرّةٍ تُعرض المكتبة للشخص (لا زرّ)، «الفتح» عند أوّل تنزيل/تشغيل، «الإنجاز» بزرّ إقرارٍ صريح «أتممتُ قراءته/سماعه» (مسؤوليّة أمانة).
- الجمهور `audience` يُفسَّر بالأدوار القائمة: amir ⇒ كلّ من له تكليف amir نشط، وهكذا.
- الرفع بواجهة الإدارة عبر `media.server.ts` القائم (R2 MEDIA) بمفاتيح `materials/<id>/<filename>`.
- **لوحة المتابعة (للإدارة والطبقات):** مصفوفة أشخاص × موادّ إلزاميّة ضمن نطاق الناظر (المدير يرى الجميع، مسؤول المنطقة يرى مساجده…) بنسب إنجاز ملوّنة + تصدير CSV.
- **تذكيرات كرون:** مادّة إلزاميّة لم تُنجَز خلال ١٤ يومًا من الاستلام ⇒ إشعار `material_reminder` أسبوعيّ (idempotent بمفتاح person|material|week) عبر القنوات الثلاث.

**اختبارات:** إسناد بالجمهور، خطّ الزمن delivered→opened→completed، عزل النطاق في المصفوفة، تذكير بلا تكرار.

### د) الدروس/المحاضرات (للمساجد) + الحضور

**هجرة `0042_lessons`:** `mosque_lessons` (id, mosque_id, title, description, place, starts_at, duration_min, material_id?, status: scheduled|confirmed|delivered|cancelled, created_by, created_at) + `mosque_lesson_attendance` (lesson_id, person_id?, name, present).
- **كشف التعارض:** عند الحفظ استعلام تداخلٍ زمنيّ لنفس المسجد ⇒ تحذير «يوجد درسٌ آخر في نفس الموعد» (غير مانع — قد يتعمّدونه بقاعتين).
- واجهة في صفحة المسجد: «جلسات اليوم» بارزة + «القادم» + «المُلقاة» + طباعة كشف (نمط PDF القائم في الشبكة).
- الحضور: كشف أسماء (طلاب الحلقات + إضافة حرّة) بنقرة، وتقرير حضور الشخص عبر الدروس.

### ن) النشاطات والمتابعة — «المطلوب اليوم» ✦ جوهر مجلس

**هجرة `0043_activities`:** `activities` (id, scope_kind: circle|mosque, scope_id, title, details, due_at, required, created_by, created_at, status) + `activity_responses` (id, activity_id, person_id, body, attachment_r2?, submitted_at, review_status: pending|seen|accepted, reviewed_by).
- شاشة الطالب «**المطلوب منّي**»: نشاطات نطاقه غير المجابة مرتّبةً بالاستحقاق + «تمّ إرسال ردّك».
- شاشة الشيخ «**متابعة الردود**»: لكلّ نشاطٍ عدّاد (أجاب/لم يجب) وقائمة الردود مع قبول.
- إشعار عند نشاط جديد (لطلاب النطاق ممّن لهم حساب) وعند ردٍّ جديد (للمنشئ).

### خ) الاختبارات والواجبات

**هجرة `0044_exams`:** `exams` (id, scope_kind, scope_id, kind: exam|homework, title, publish_at, due_at, status, created_by) + `exam_questions` (id, exam_id, sort, kind: mcq|tf, text, options JSON, correct, points) + `exam_submissions` (id, exam_id, person_id, answers JSON, score, submitted_at, UNIQUE(exam_id, person_id)).
- تصحيح آليّ للاختيار من متعدّد وصح/خطأ؛ «انتهى وقت التسليم» يقفل التسليم؛ الدرجات تظهر للطالب وللشيخ.

### ع) العُهدة والأصول + المركبات والمحروقات

**هجرة `0045_assets`:** `assets` (id, kind: personal_custody|vehicle|equipment, name, details, org_unit_id, holder_person_id?, status: active|returned|retired, created_at) + `asset_expenses` (id, asset_id, month TEXT 'YYYY-MM', fuel_amount REAL, other_amount REAL, note) — «ليكون عندنا تقييمٌ لكلّ شيء»: تقرير شهريّ لمصروف المحروقات لكلّ مركبةٍ ولكلّ وحدة.

### ذ) محرّك التذكيرات النسبيّة + الإعلانات

- تعميم خطوة الكرون: تذكير درسٍ «قبل ٣ ساعات» (قابل للضبط)، نشاطٌ يستحقّ «بعد يوم»، اختبارٌ يُغلق «قبل يوم» — جدول `reminder_rules` بسيط أو ثوابت مضبوطة أوّلًا.
- إعلانات المنصّة: `announcements` (نطاق، نصّ، تاريخ) تظهر في الجرس للجميع ضمن النطاق.
- إعدادات إشعاراتٍ للشخص (كتم أنواع) — مرحلة لاحقة إن لزم.
