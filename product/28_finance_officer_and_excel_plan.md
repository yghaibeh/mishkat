# ٢٨ — خطّة «المسؤول الماليّ بالاعتماد الثنائيّ» و«الاستيراد/التصدير الشامل Excel»

> **الحالة: نُفِّذت كاملةً ونُشرت (٢٠٢٦-٠٧-١٤)** — الدفعات د١–د٥ كلُّها، هجرتا 0070+0071 على remote، ٢٩٧/٢٩٧ اختبارًا، آخر إصدار **63615500**. التفصيلُ والانحرافاتُ الموثَّقة (ExcelJS بدل SheetJS CE لدعم القوائم المنسدلة وسياسة npm؛ payout بقي admin-only؛ المثالُ في ورقة التعليمات) في [سجلّ التنفيذ](06_implementation_log.md).


> **وثيقةُ خطّةٍ هندسيّةٍ تفصيليّة — لا تنفيذَ فيها.** كُتبت بطلب المالك (٢٠٢٦-٠٧-١٤):
> «نريد خاصية استقبال وتصدير من ملف اكسل ليكون الملف شاملًا كاملًا.. وإمكانية إضافة أخ صفته "مسؤول مالي"
> يتحكّم بهذا القسم والمدير يعتمد عمله في كلّ خطوة يخطوها.. ميزة ضخمة تتصل بكلّ خصائص النظام — اكتب خطة دقيقة جدًا جدًا».

## ٠) الخلاصة التنفيذيّة

| الميزة | الجوهر | البنية |
|---|---|---|
| **أ. المسؤول الماليّ** | دورٌ جديد `finance_officer` يعمل بكامل القسم الماليّ، لكن **كلَّ فعلٍ يمسّ المال/الدفتر لا يُنفَّذ إلا باعتماد المدير** | **محرّكُ اعتمادٍ ثنائيّ (Dual-Control Engine)**: طابورُ «أفعالٍ ماليّةٍ معلّقة» + نقطةُ اعتراضٍ واحدة + سجلُّ منفّذين يعيد استخدام كلّ الخدمات القائمة |
| **ب. Excel** | تصديرُ **مصنّفٍ واحدٍ شامل** (١٩ ورقة) + استيرادٌ **بقوالبَ مُتحقَّقٍ منها** يمرّ هو نفسُه بالاعتماد | توليدٌ في المتصفّح (SheetJS) + خطُّ استيرادٍ مرحليّ (تحليل ← تحقّق ← معاينة ← اعتماد ← تنفيذٌ مجزّأٌ قابلٌ للاستئناف) |

**هجرتان فقط** (0070 محرّك الاعتماد، 0071 دفعات الاستيراد). **لا تغييرَ على محرّك الدفتر** — القيدُ المزدوج والتوازنُ والعكسُ بدل الحذف كلُّها كما هي.

## ١) لماذا «محرّك» لا «شرط if»؟ — المبادئ المعماريّة الحاكمة

1. **نقطةُ اختناقٍ واحدة (Single Choke Point):** كلُّ فعلٍ ماليٍّ للمسؤول الماليّ يمرّ عبر دالّةٍ واحدة `guardFinanceAction(kind, payload)`. لا نكرّر منطقَ «هل يحتاج اعتمادًا؟» في ٢٠ مكانًا — هذا هو الفرق بين الحلّ العالميّ والترقيع.
2. **إعادةُ استخدام الخدمات القائمة حرفيًّا:** المُنفِّذون (executors) هم دوالُّ الخدمات الموجودة (`recordDonation`, `recordExchange`, `payBatch`, …) — المحرّك يؤجّل استدعاءَها فقط. لا ازدواجَ منطقٍ إطلاقًا.
3. **الاعتمادُ عند أثر المال فقط (افتراضيًّا):** المسودّاتُ الحرّة (إنشاء دفعةٍ فارغة، إضافة بند، تسجيل تعهّد) لا تحتاج اعتمادًا لأنها لا تمسّ الدفتر؛ **أيُّ ترحيلٍ للدفتر أو حركةِ مالٍ يحتاجه**. (يوجد «وضعٌ صارمٌ» اختياريّ يُخضِع حتى المسودّات — قرار مفتوح §١٠).
4. **قدسيّةُ الدفتر محفوظة:** المحرّك يعمل *قبل* `postJournal` لا بعده. لا قيودَ «معلّقة» داخل الدفتر — الدفترُ لا يحوي إلا حقائقَ معتمَدة.
5. **Idempotency في كلّ طبقة:** فعلٌ معلّقٌ له `clientUuid`؛ استيرادٌ له `contentHash`؛ كلُّ صفّ استيرادٍ له مفتاحٌ حتميّ — إعادةُ الإرسال/الرفع آمنةٌ دومًا.
6. **الملفُّ لا يُنفَّذ أبدًا مباشرة:** الاستيرادُ = اقتراحُ دفعةٍ تمرّ بنفس محرّك الاعتماد. Excel قناةُ إدخالٍ، لا بابًا خلفيًّا.
7. **الأرقام تُولَّد من مصدر الحقيقة:** التصديرُ يقرأ من نفس دوالّ الخادم التي تُغذّي الشاشات (نفسُ العزل بالنطاق والصلاحيّات) — لا استعلاماتِ تصديرٍ موازية تنحرف عن الشاشة.

## ٢) الميزة أ — دورُ «المسؤول الماليّ» ومحرّكُ الاعتماد الثنائيّ

### ٢.١ الدور والصلاحيّات

| عنصر | القيمة |
|---|---|
| معرّف الدور | `finance_officer` — «مسؤول ماليّ» / «مسؤولة ماليّة» (تأنيثٌ كالمعتاد) |
| قدراته الافتراضيّة | `finance.view` + `finance.entry` + `finance.payout` + `duties.view` + `library.view` — **بلا** `finance.approve` (لا يعتمد شيئًا أبدًا) |
| قدرةٌ جديدة للمدير | `finance.supervise` — «اعتمادُ أعمال المسؤول الماليّ» (الإدارة العليا بـ«*»؛ قابلةٌ للمنح لاحقًا لنائبٍ عبر لوحة الصلاحيّات القائمة) |
| التوفير | من **لوحة المستخدمين القائمة** (اسم + دخول + كلمة مرور + نطاق) — نفس بنية `provisionUser` المستخدمة للمعلّم ومسؤول اللجنة، بلا كود جديد |
| النطاق | يُكلَّف على الجذر/القسم فيغطّي الماليّةَ المركزيّةَ كاملةً؛ العزلُ القائم بالمسارات يسري تلقائيًّا |

**قاعدةُ الفصل الذهبيّة (invariant):** حاملُ `finance_officer` تُعترَض كلُّ أفعاله الماليّة؛ ولا يستطيع اعتمادَ أيّ فعلٍ (ولا فعلِه هو) حتى لو نودي الخادمُ مباشرةً — تُثبتها اختباراتٌ صريحة.

### ٢.٢ دورةُ حياة الفعل الماليّ (State Machine)

```
المسؤول الماليّ يضغط «حفظ» في أيّ نموذجٍ ماليّ
        ↓ (اعتراضٌ شفّاف — لا رسالة خطأ)
   [pending معلّق] ──── المسؤول يلغيه ────→ [cancelled مُلغى]
        ↓ المدير يقرّر من «صندوق الاعتماد الماليّ»
   ┌────┴─────────┐
[approved معتمَد]  [rejected مرفوض + سببٌ إلزاميّ] → المسؤول يعدّل ويعيد الإرسال (فعلٌ جديدٌ يشير للقديم)
        ↓ تنفيذٌ فوريٌّ ذرّيّ (استدعاء الخدمة القائمة)
   ┌────┴─────────┐
[executed منفَّذ]   [failed فشل التنفيذ + الخطأ] → إعادةُ محاولةٍ للمدير (مثال: الفترةُ أُقفلت بين الاقتراح والاعتماد)
```

- **الحمولة تُجمَّد** عند الإرسال (JSON لمدخلات الدالّة حرفيًّا) — التعديل = إلغاءٌ + اقتراحٌ جديد (تاريخٌ نظيفٌ للتدقيق).
- كلُّ انتقالٍ يسجَّل في `audit_log` القائم (مَن/متى/ماذا).

### ٢.٤ طبقةُ الاعتراض وسجلُّ المنفّذين — **تغطيةُ كلّ الأفعال**

نقطةُ الاختناق في الخادم:

```ts
// داخل كلّ server-fn ماليٍّ مُغيِّر — سطرٌ واحد:
const gate = await guardFinanceAction(u, "donation_add", input, summaryOf(input));
if (gate.queued) return { queued: true, actionId: gate.id }; // اعتُرض ⇒ للطابور
// وإلا يتابع التنفيذَ المباشر كما اليوم
```

### ٢.٥ معاينةُ أثر الدفتر (Dry-Run) — قبل الاعتماد

الميزةُ التي تجعل الاعتمادَ حقيقيًّا لا شكليًّا: **قبل أن يعتمد المديرُ يرى القيدَ الذي سيُرحَّل** (مدين/دائن بأسماء الحسابات والصندوق والعملة). تنفيذًا: تُستخرج «بُناةُ السطور» من `ledgerPost`/الخدمات (هي حساباتٌ خالصةٌ قبل `postJournal`) إلى دوالَّ نقيّةٍ `previewLines(kind, payload)` تُستدعى للعرض دون كتابة. الفوائد: صفرُ مفاجآت، وتعليمٌ ضمنيٌّ للمدير غير المحاسب.

### ٢.٦ الواجهات

1. **للمسؤول الماليّ (نفس صفحة الماليّة):**
   - كلُّ زرِّ حفظٍ خاضعٍ يتحوّل نصُّه تلقائيًّا إلى **«إرسالٌ للاعتماد»** (يعرف ذلك من `gate.queued` — بلا شرطيّاتٍ يدويّةٍ في كلّ نموذج: مكوّنٌ مغلّفٌ واحد).
   - قسمٌ جديد أعلى الصفحة: **«مقترحاتي»** — المعلّق (مع إلغاء)، المرفوض (السبب + «تعديلٌ وإعادةُ إرسال» يملأ النموذجَ بالحمولة)، المنفَّذ حديثًا.
2. **للمدير:**
   - **«صندوقُ الاعتماد الماليّ»** — قسمٌ بارزٌ أعلى صفحة الماليّة (نمطُ «بانتظار اعتمادك» القائم): بطاقةٌ لكلّ فعلٍ (الملخّص + المقترِح + الزمن + المبلغ) تتوسّع إلى **التفاصيل الكاملة + معاينة القيد** + زرّا «اعتماد» / «لا يُعتمد (بسببٍ إلزاميّ)».
   - تجميعٌ بالنوع + **اعتمادٌ جماعيٌّ** للمتشابه الصغير (اختياريٌّ بتأكيد).
   - عدّادٌ في جرس الإشعارات وفي «المطلوب اليوم».
3. **سجلُّ الاعتمادات** (تبويبٌ للمطّلِع): كلُّ الأفعال بتاريخها وحالتها ومقرِّرها — مرشَّحٌ بالنوع/الحالة/الفترة، وهو نفسُه ورقةٌ في تصدير Excel.
4. **مؤشّرات:** معلّقٌ الآن، مُعتمَد هذا الشهر، متوسّطُ زمن الاعتماد، نسبةُ الرفض.

### ٢.٧ الإشعارات (القنوات القائمة: جرس + تيليغرام + Push)

| حدث | إلى | القناة |
|---|---|---|
| اقتراحٌ جديد | حاملي `finance.supervise` | جرس + تيليغرام |
| اعتماد/رفض | المقترِح | جرس + تيليغرام |
| معلّقٌ > ٤٨ ساعة | المدير | تذكيرٌ عبر الكرون الساعيّ القائم |
| فشلُ تنفيذٍ بعد اعتماد | المدير + المقترِح | جرس |

### ٢.٨ ثوابتُ الأمن والفصل (تُثبَّت باختبارات)

1. المسؤول الماليّ **لا يملك** أيَّ مسارِ تنفيذٍ مباشرٍ لفعلٍ خاضع (اختبارُ استدعاءِ server-fn مباشرةً ⇒ queued لا executed).
2. لا يعتمد أحدٌ فعلَ نفسِه (حتى لو جمع الدورين يومًا).
3. الرفضُ بلا سببٍ مرفوض.
4. التنفيذُ بعد الاعتماد يحترم كلَّ الحرّاس القائمين (فترةٌ مقفلة/صندوقٌ مقيَّد/رصيدُ نثريّة…) — الفشلُ يُسجَّل ولا يُبتلَع.
5. تكرارُ الاعتماد على نفس الفعل لا يزدوج (idempotent).
6. الدفترُ لا يحوي أثرًا لفعلٍ غير معتمَد (Σمدين=Σدائن قبل وبعد، واختبارُ غياب القيد).

## ٣) الميزة ب — الاستيراد والتصدير Excel

### ٣.١ قرارُ التقنية

- **التوليد والقراءة في المتصفّح** بمكتبة **SheetJS CE (`xlsx`، رخصة Apache-2.0)** مُحمَّلةً كسولًا (~٤٠٠KB gz عند أوّل استخدامٍ فقط):
  - يتجنّب حدَّ CPU في Workers (50ms) لملفّاتٍ كبيرة.
  - البياناتُ تصل JSON من دوالّ الخادم القائمة (نفسُ الصلاحيّات والعزل) — **المتصفّحُ يصوغ الملفَّ فقط**.
  - الاستيراد: المتصفّح يحلّل الملفَّ إلى صفوف JSON ويرسلها — الخادمُ لا يلمس ملفّاتِ xlsx إطلاقًا (سطحُ هجومٍ أصغر).
- RTL (`views:[{rightToLeftView:true}]`)، عناوينُ عربيّة، صفُّ رؤوسٍ مجمّد، عرضُ أعمدةٍ مضبوط، تنسيقُ أرقامٍ `#,##0.00`، تاريخان (هجريّ نصًّا + ميلاديّ).

### ٣.٢ التصدير الشامل — مصنّفٌ واحدٌ «شامل كامل» (١٩ ورقة)

زرّ **«تصدير Excel شامل»** في رأس صفحة الماليّة، مع مرشّحِ فترةٍ (سنة/شهر هجريّ) قبل التوليد. دالّةُ خادمٍ واحدةٌ مجمِّعة `financeWorkbookData(period)` (تستدعي الدوالَّ القائمة) لتقليل الجولات.

| # | الورقة | الأعمدة الرئيسة |
|---|---|---|
| ١ | **الملخّص** | أرصدةُ الصناديق الخمسة، النقدُ بكلّ عملة (أصليّ + دولاريّ)، إجماليّات الفترة، عددُ الحركات، تاريخُ التوليد وبصمةُ الإصدار |
| ٢ | **دفتر اليوميّة** | التاريخ (هـ/م)، البيان، المصدر، الحساب، الصندوق، مدين، دائن، العملة الأصليّة ومبلغُها، المرجع |
| ٣ | **ميزان المراجعة** | الحساب (رمزٌ واسم)، النوع، مدين، دائن، الرصيد |
| ٤ | **قائمة النشاط** | لكلّ صندوق: إيراد، مصروف، صافي التغيّر + الإجماليّ |
| ٥ | **المركز الماليّ** | الأصول، الخصوم، صافي الأصول بالصندوق + برهانُ التوازن |
| ٦ | **التدفّق النقديّ** | تشغيليّ/استثماريّ/تمويليّ ببنودها + صافي التغيّر + الرصيد |
| ٧ | **التبرّعات** | السند، التاريخ، المانح، الصندوق، المبلغ (أصليّ + $)، المسجد، الملاحظة |
| ٨ | **المصروفات** | التاريخ، البند، الصندوق، المبلغ (أصليّ + $)، المسجد، المسجِّل |
| ٩ | **المانحون والتعهّدات** | المانح، إجماليُّ عطائه، تعهّداتُه (الموفى/المتبقّي/الصندوق) |
| ١٠ | **الموازنات** | الفترة، الصندوق، المخطّط، الفعليّ، الانحراف، ٪ |
| ١١ | **مطالبات الصرف** | التاريخ، البند، الصندوق، المبلغ، الحالة، الطالب، المقرِّر، سببُ الرفض، رابطُ الإيصال |
| ١٢ | **الرواتب** (بشهر) | الشخص، الإجماليّ، البدلات، الخصومات، قسطُ السلفة، الصافي، الحالة |
| ١٣ | **السُلَف** | الشخص، الأصل، القسط، المسدَّد، المتبقّي، الحالة |
| ١٤ | **النثريّة** | الصناديق (سقف/رصيد/منصرف) + ورقةُ حركاتها |
| ١٥ | **دفعات الصرف** | الدفعة، الحالة، البنود (مستفيد/مبلغ)، الإجماليّ، مقرِّرُ الصرف |
| ١٦ | **الأصول الثابتة** | الأصل، التكلفة، المتبقّية، العمر، القسط، المجمّع، **القيمة الدفتريّة**، الحالة + جدولُ إهلاكِ كلّ أصل |
| ١٧ | **العملات والتصريف** | العملات وأسعارُها التاريخيّة + عمليّاتُ التصريف بمكاسبها/خسائرها |
| ١٨ | **المطابقة البنكيّة** | لكلّ حساب نقد: رصيدُ الدفتر، المُطابَق، غيرُ المُطابَق |
| ١٩ | **سجلّ الاعتمادات** | الفعل، الملخّص، المقترِح، القرار، المقرِّر، الزمن، السبب |

- **حمايةُ حقن الصيغ (CSV/Excel Injection):** أيُّ خليّةٍ نصّيّةٍ تبدأ بـ `=`، `+`، `-`، `@` تُسبَق بفاصلةٍ عليا `'` — تصديرًا واستيرادًا (تُطبَّق أيضًا رجعيًّا على تصدير CSV القائم).
- **عزلُ الحساسية:** أوراقُ الرواتب/السُلَف تُضمَّن فقط لمن يملك صلاحيّتها (نفسُ منطق الشاشة).

### ٣.٣ الاستيراد — القوالبُ المعتمدة (لا ملفّاتَ حرّة)

زرّ **«تنزيل قالب»** لكلّ نوع — قالبٌ فيه: صفُّ رؤوسٍ ثابتٍ (مفاتيحُ إنجليزيّةٌ مخفيّةٌ في صفٍّ ثانٍ + عناوينُ عربيّة)، قوائمُ تحقّقٍ منسدلة (الصناديق/العملات/الأنواع)، صفُّ مثال، وورقةُ «تعليمات».

| قالب | الأعمدة | تحقّقاتُ الخادم الأساسيّة |
|---|---|---|
| **تبرّعات** | التاريخ (هـ أو م)، المانح، الصندوق، المبلغ، العملة، المسجد (رمز/اسم)، ملاحظة | صندوقٌ معروف، عملةٌ لها سعر، مبلغٌ موجب، مسجدٌ ضمن النطاق، كشفُ تكرارٍ (تاريخ+مانح+مبلغ) |
| **مصروفات** | التاريخ، البند، الصندوق، المبلغ، العملة، المسجد، ملاحظة | + حارسُ الصندوق المقيَّد يعمل وقتَ التنفيذ |
| **أرصدة افتتاحيّة** *(ميزة مضافة)* | الحساب (1110/1115/1120/1130)، الصندوق، المبلغ، العملة | مرّةٌ واحدةٌ لكلّ (حساب،صندوق) — قيد Dr نقد / Cr 3100 الرصيد المرحَّل |
| **بنود دفعة صرف** | اسم المستفيد، المبلغ، ملاحظة | تُلحق بدفعةٍ مسودّةٍ يختارها |
| **تعديلات رواتب** | الشخص (دخول/اسم)، الشهر، النوع (بدل/خصم)، المبلغ، البيان | شخصٌ موجود، شهرٌ صالح |
| **أسعار صرف** | العملة، السعر، تاريخ السريان | عملةٌ غيرُ الأساس، سعرٌ موجب |
| **موازنات** | الفترة، الصندوق، المبلغ | صيغةُ الفترة `1447` أو `1447-12` |
| **أصول ثابتة** | الاسم، التكلفة، المتبقّية، العمر بالأشهر، شهرُ البدء، الصندوق | متبقّية < تكلفة، عمرٌ صحيحٌ موجب |
| **مانحون** | الاسم، الهاتف، ملاحظة | كشفُ تكرار الاسم |

### ٣.٤ خطُّ الأنابيب المرحليّ + هجرة **0071**

```
(١) رفعٌ في المتصفّح ⇒ تحليلُ SheetJS ⇒ صفوف JSON (سقف: ٢MB / ٢٠٠٠ صفّ)
(٢) POST validateImport ⇒ تحقّقٌ صفًّا صفًّا (zod لكلّ نوع + مراجعُ الوجود + التكرار)
(٣) «معاينة»: ٟn صالح / n تحذير / n خطأ + جدولُ الأخطاء بالسطر + تنزيلُ «ورقة أخطاء» لإصلاحها وإعادة الرفع
     السياسة الافتراضيّة: الكلُّ أو لا شيء (لا يُقبل ملفٌّ فيه خطأ) — تشدّدٌ محاسبيٌّ مقصود
(٤) تأكيدُ المُستورِد ⇒ يُنشأ import_batch + صفوفُه ⇒ **فعلُ bulk_import في محرّك الاعتماد** (المدير يرى: النوع، العدد، الإجماليّ، عيّنة)
(٥) اعتمادُ المدير ⇒ تنفيذٌ مُجزّأ (شرائح ≤٢٥ صفًّا لكلّ db.batch احترامًا لحدود D1) بمؤشّر تقدّمٍ قابلٍ للاستئناف عند أيّ انقطاع
(٦) كلُّ صفٍّ يُنفَّذ عبر الخدمة القائمة نفسِها بمفتاح idempotency حتميّ = hash(batchId+rowNo) ⇒ إعادةُ التنفيذ لا تزدوج أبدًا
```

**`import_batches`** (0071): id، kind، filename، content_hash (كشفُ رفع نفس الملفّ مرّتين)، row_count، valid/warn/error counts، status (validating / awaiting_approval / approved / executing / done / failed / rejected)، executed_rows (للاستئناف)، proposed_by/decided_by/timestamps، finance_action_id.
**`import_rows`**: id، batch_id، row_no، payload JSON، status (valid/warn/error/executed)، message.

### ٣.٥ سلامةُ الاستيراد

سقفُ الحجم والصفوف · تعقيمُ الخلايا من حقن الصيغ · لا صيغَ تُقرأ (قيمٌ فقط) · صلاحيّةُ `finance.entry` للرفع + **اعتمادٌ إلزاميٌّ دائمًا** للتنفيذ (حتى لو رفعه المدير نفسُه تُعرض عليه المعاينة النهائيّة) · content_hash يمنع الدفعةَ المكرّرة · سجلُّ تدقيقٍ كامل.

## ٤) ميزاتٌ عالميّةٌ مُضافة (ضمن الخطّة)

1. **معاينةُ القيد قبل الاعتماد** (§٢.٥) — قلبُ جودة القرار.
2. **قيدٌ يدويٌّ مُعتمَد (Manual Journal):** نموذجٌ متقدّمٌ (سطور مدين/دائن بحساب وصندوق) للحالات النادرة — يمرّ حتمًا بالاعتماد بمعاينة. يُغلق آخرَ فجوةٍ بين النظام وبرامج المحاسبة الكاملة.
3. **الأرصدة الافتتاحيّة** عبر قالبٍ مخصّص — شرطُ أيّ انتقالٍ من دفاترَ ورقيّة.
4. **عتباتُ اعتمادٍ قابلةٌ للضبط** (`approval_policies.threshold_usd`) — اليوم ٠ (كلُّ شيء)، وغدًا «النثريّةُ دون ٢٠$ مباشرة» بقرارٍ إداريّ دون كود.
5. **نائبُ معتمِد:** منحُ `finance.supervise` لدورٍ آخر من لوحة الصلاحيّات القائمة — بلا كودٍ إضافيّ.
6. **كشفُ تكرار الملفّات** (content_hash) و**ورقةُ الأخطاء** القابلةُ للإصلاح وإعادة الرفع.
7. **مؤشّراتُ الاعتماد** (زمنُ الاستجابة/نسبةُ الرفض) — حوكمةٌ قابلةٌ للقياس.

## ٦) معايير القبول النهائيّة

1. مسؤولٌ ماليٌّ يسجّل تبرّعًا ⇒ **لا شيء في الدفتر**، ويظهر معلّقًا عنده وفي صندوق المدير بإشعار.
2. المدير يفتح البطاقة فيرى التفاصيل **ومعاينةَ القيد** (Dr نقد / Cr تبرّعات بصندوقه) ⇒ يعتمد ⇒ يُرحَّل فورًا ويُشعَر المقترِح، وسندُ القبض يصدر حينها فقط.
3. الرفضُ يعود بسببٍ ظاهرٍ للمسؤول مع زرّ «تعديل وإعادة إرسال».
4. استدعاءُ أيّ server-fn ماليٍّ مباشرةً بحساب المسؤول ⇒ `queued` حتمًا (اختبارٌ آليّ).
5. «تصدير Excel شامل» ⇒ ملفٌّ واحدٌ بـ١٩ ورقةً عربيّةً RTL أرقامُها تطابق الشاشةَ بالسنت.
6. استيرادُ قالبِ تبرّعاتٍ فيه صفٌّ فاسد ⇒ لا يدخل شيءٌ، وتقريرُ أخطاءٍ بالسطر + ورقةُ أخطاءٍ قابلةٌ للتنزيل.
7. ملفٌّ صالح ⇒ يظهر للمدير كفعلِ استيرادٍ واحدٍ بعدده وإجماليّه ⇒ اعتمادُه ينفّذ الصفوفَ كلَّها (مع استئنافٍ عند الانقطاع) ⇒ إعادةُ رفع نفس الملفّ تُرفَض (hash).
8. Σمدين=Σدائن والمركزُ متوازنٌ بعد كلّ سيناريو E2E.

## ٧) المخاطر والتخفيف

| خطر | تخفيف |
|---|---|
| اختناقُ المدير بكثرة الاعتمادات | اعتمادٌ جماعيّ + عتباتٌ قابلةُ الضبط + نائبٌ عبر الصلاحيّات + تذكيرات |
| تغيّرُ الواقع بين الاقتراح والاعتماد (سعرُ صرفٍ/فترةٌ أُقفلت) | التنفيذُ يعيد كلَّ التحقّقات وقتَه؛ الفشلُ يُعرض بسببه لا يُبتلَع |
| حدودُ D1 (متغيّرات/زمن) في الاستيراد | شرائح ≤٢٥ + تتبّعُ executed_rows + استئناف |
| ثقلُ SheetJS | تحميلٌ كسولٌ عند أوّل تصدير/استيراد فقط |
| حقنُ الصيغ | تعقيمٌ إلزاميٌّ تصديرًا واستيرادًا (يشمل CSV القائم) |
| التفافٌ على الاعتماد عبر واجهات المساجد | نطاقُ الدور والسياسة بالدور لا بالصفحة — أينما نفذ فعلُه اعتُرض |

## ٨) القراراتُ المعتمدة (فوّض المالكُ الاعتمادَ مع التوثيق — ٢٠٢٦-٠٧-١٤)

1. ✅ **مدى «كل خطوة»:** كلُّ ما يمسّ المال/الدفتر يُعتمد؛ المسودّاتُ الحرّة (بنودُ دفعةٍ قبل صرفها، تقديمُ مطالبة، تعهّد) لا تحتاج اعتمادًا. **السبب:** الاعتمادُ حيث الأثرُ الماليّ يوازن الضبطَ والسيولةَ التشغيليّة؛ الوضعُ الصارمُ متاحٌ لاحقًا بسياسةٍ (`approval_policies`) دون كود.
2. ✅ **أفعالُ الأمراء** على ماليّة مساجدهم تبقى **مباشرةً** كما اليوم. **السبب:** لا سياسةَ لدور amir؛ إخضاعُه لاحقًا = صفُّ بذرةٍ واحدٌ في الجدول.
3. ✅ **وسمُ المطابقة البنكيّة بلا اعتماد** — لا يحرّك مالًا وهو قابلٌ للعكس، وإخضاعُه يعرقل الجردَ اليوميّ.
4. ✅ **المعتمِد: المديرُ وحده** بدايةً (`finance.supervise` عبر «*»)؛ تعيينُ نائبٍ لاحقًا = منحُ القدرة من لوحة الصلاحيّات القائمة بلا كود.
5. ✅ **أوراقُ الرواتب/السُلَف في التصدير حسب صلاحيّة الطالب** (نفسُ منطق الشاشات).
6. ✅ **قرارٌ مُلحق — استيرادُ المدير نفسِه:** ثابتُ «لا يعتمد أحدٌ فعلَ نفسه» مُقدَّمٌ على قاعدة «الاستيراد يُعتمَد دائمًا»؛ فاستيرادُ حامل `finance.supervise` يُنفَّذ بعد **معاينته الإلزاميّة وتأكيده الصريح** (المعاينةُ هي اعتمادُه)، واستيرادُ المسؤول الماليّ يلزمه اعتمادُ المدير حتمًا.
7. ✅ **إعادةُ الإرسال بعد الرفض (v1):** يُعرض السببُ وزرٌّ يعيد فتحَ النموذج ببياناته؛ لا نسخَ سلاسلِ تعديلٍ معقّدة في الإصدار الأوّل (العمود `resubmit_of` جاهزٌ للاحق).

## ٩) ما لن يتغيّر (ضمانات)

محرّكُ الدفتر (توازن/عكس/idempotency/فترات) كما هو حرفًا · تدفّقاتُ الأمراء والمعلّمين الحاليّة لا تُمسّ · كلُّ الاختبارات القائمة (٢٦٧) تبقى خضراء · لا حذفَ ولا تعديلَ لقيدٍ — أبدًا.
