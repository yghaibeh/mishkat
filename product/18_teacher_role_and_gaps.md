# الجزء الثاني: خطة دمج دور «مدرّس/محفّظ» بشكل تامّ ١٠٠٪

## 1) المشكلة والمتطلّبات (من المالك)
- دور جديد: **مدرّس/محفّظ**، له **تقرير بما درّسه + عدد طلابه**.
- توجد **حلقات نسائية كاملة** أو **حلقات تحفيظ فقط** **بلا أمير مسجد** — على النظام قبولها.
- المدرّس له **فعلياً كل ما في «الحلقة الدورية»** (الدروس/الطلاب/التقييم/الملاحظات الأسبوعية…).
- **لا حلول التفافية** — دمج جذري من جميع الجوانب.

## 2) الحقيقة المعمارية الحالية (نتيجة التدقيق)
ثلاثة مفاهيم «حلقة»:
- `circles` + `circle_students` — **مرتبطة بمسجد إجبارياً** (`mosqueId NOT NULL`)، وصفية بلا دروس/مالية.
- `tahfeez_circles` + `tahfeez_students` — **مرتبطة بمسجد إجبارياً**؛ فحص النطاق يشترط **أمير المسجد** صراحةً.
- **«على بصيرة»**: `venues`(orgUnitId **اختياري/قابل null**) → `halaqat` → `enrollments` + `lesson_sessions` + `student_evaluations` + التقييم الأسبوعي، و**جدول `teachers`** (شخص + سعر ساعة)، و**الدروس تغذّي المالية** (ساعات × سعر).

**الخلاصة الحاسمة:** «على بصيرة» **هو فعلياً «الحلقة الدورية»** المطلوبة، وهو الوحيد الذي يدعم أصلاً اللاارتباط بأمير، وله كل الميزات (دروس/طلاب/تقييم/ملاحظات/مالية). لذا **نبني الدور فوق نموذج «على بصيرة» لا فوق `circles`/`tahfeez` المقيّدة بالمسجد** — هذا هو الدمج الجذري بلا التفاف.

## 3) قرار التصميم الجوهري (موصى به)
**المدرّس = مالكُ حلقاتٍ من نوع «على بصيرة»، مرتبطة بوحدةٍ إشرافية (مربع/منطقة) لا بمسجدٍ بالضرورة.**
- كل حلقة لها `teacherId` (موجود) → `teachers.personId` → حساب المدرّس.
- مكان الحلقة (`venue.orgUnitId`) يشير إلى **الوحدة المشرفة** (غالباً مربع، أو مسجد إن وُجد) — فيبقى المشرفون يرونها ضمن نطاقهم بلا أمير.
- التحفيظ يُمثَّل كحلقة على بصيرة بنوعٍ مميّز (`halaqat.kind = 'tahfeez' | 'baseera' | 'general'`) — فيستفيد من نفس الدروس/الطلاب/المتابعة دون جدولٍ مكرّر.

> هذا يلغي الحاجة لجعل `circles`/`tahfeez_circles` بلا مسجد، ويمنع تشظّي ثلاثة نماذج. (انظر القرار D1 أدناه — بديلٌ مطروح.)

## 4) تغييرات قاعدة البيانات (هجرات جديدة)
1. `teachers`: إضافة عمود **`userLinked`/استخدام `personId` الموجود** كهوية الدخول (لا تغيير بنيوي — `personId` كافٍ).
2. `halaqat`: إضافة `kind TEXT DEFAULT 'baseera'` (baseera | tahfeez | general) لتمييز نوع الحلقة.
3. (إن لزم) `venues.orgUnitId` يبقى nullable — **لا تغيير**.
4. لا تغيير على `role_assignments` (يقبل أي `role`).

## 5) RBAC — إضافة الدور
- `src/server/utils/rbac.ts`: `TEACHER: 'teacher'`.
- `src/lib/capabilities.ts`:
  - `ROLE_LABEL.teacher = "مدرّس/محفّظ"`.
  - قدرات جديدة: `circle.teach` (إدارة حلقاته فقط) — أو إعادة استخدام `alaBaseera.view` + قدرة ملكية.
  - `ROLE_DEFAULTS.teacher = ["alaBaseera.view", "circle.teach", "tahfeez.view"]` (عرض حلقاته + إدارتها، بلا مالية/شبكة/تهيئة).
- تكليف المدرّس: `role_assignments(role='teacher', orgUnitId=الوحدة المشرفة, orgPath=مسارها)` — فيُحصر نطاقه.

## 6) فحوص النطاق (الجذر — لا التفاف)
- دالة جديدة `isTeacherOfHalaqa(u, halaqaId)`: `halaqat.teacherId → teachers.personId === u.personId`.
- تعديل `halaqaInScope` و`requireAlaBáseeraManage`:
  «admin **أو** أمير الوحدة المالكة **أو** المدرّس المالك للحلقة».
- المدرّس **لا** يصل لحلقات غيره (عزل صارم بالملكية + النطاق).
- المشرف (مربع/منطقة) يرى حلقات نطاقه عبر `venue.orgUnitId` ضمن `orgPath` (قائم).

## 7) الهوية والملاحة (تجربة المدرّس)
- عند الدخول: المدرّس **لا يهبط على مسجد**؛ يهبط على **«حلقاتي»** (لوحة المدرّس).
  - تعديل `auth.server`: إن كان الدور `teacher`، `homeMosqueId=null` و`firstAllowed` يُرجِع مساراً جديداً `/my-circles`.
  - شريط علوي: تبويب «حلقاتي» (مثل بقية الشاشات العامة عبر `access.ts NAV` بقدرة `circle.teach`).
- صفحة `/my-circles`: قائمة حلقاته + لكل حلقة: الطلاب، الدروس، التقييم الأسبوعي (نفس `WeeklyHalaqaPanel` المُنجز)، وزر تسجيل درس.

## 8) التقرير (مطلب صريح: «ما درّسه + عدد طلابه»)
لوحة/تقرير المدرّس (شهري هجري):
- **عدد الطلاب** (enrollments النشطة) لكل حلقة وإجمالاً.
- **ما درّسه**: قائمة `lesson_sessions` (العنوان/المجلس/التاريخ الهجري/المدة) + **إجمالي الساعات**.
- **التقييمات** المسجّلة + متابعة الحفظ (يتكامل مع F4).
- يظهر هذا التقرير أيضاً للمشرف ضمن نطاقه (قراءة).

## 9) المالية
- ساعات دروس «على بصيرة» **تغذّي المالية أصلاً** (سعر الساعة) — المدرّس مستفيدٌ موجود بالنموذج.
- قرار: هل حلقات التحفيظ (`kind='tahfeez'`) تُحتسَب ساعاتها مالياً مثل بصيرة؟ (D3 أدناه).

## 10) البذور والاختبار
- بذرة: مدرّس (`p-teacher` + user `teacher/mishkat123` + role_assignment إلى مربع) يملك حلقة نسائية بلا مسجد + حلقة تحفيظ، بطلاب ودروس.
- اختبار e2e: دخول المدرّس → «حلقاتي» → يرى حلقتيه فقط (عزل) → يسجّل درساً + تقييماً → التقرير يعكسها → المشرف يراها ضمن نطاقه → الأمير/مدرّس آخر لا يراها.

## 11) خطة التنفيذ المرحلية
- **م١:** RBAC (الدور + القدرات) + هجرة `halaqat.kind` + فحوص الملكية.
- **م٢:** ملاحة المدرّس + صفحة «حلقاتي» (إعادة استخدام مكوّنات على بصيرة).
- **م٣:** تقرير المدرّس + متابعة التحفيظ (F4).
- **م٤:** بذور + اختبار e2e شامل + توثيق.

## 12) القرارات (مُعتمَدة من المالك ٢٠٢٦-٠٦-٣٠)
- **D1 ✅ — الجوهر:** نبني على **الركيزة الغنيّة لـ«على بصيرة»** (حلقات + دروس + طلاب + تقييم + ملاحظات أسبوعية)، **لكن مع بُعد «منهج/نوع»** لأنّ الحلقات ليست بصيرة فقط: **«على بصيرة» · «تحفيظ قرآن» · «منهج الرشيدي»** (قابلٌ للتوسّع). حقل `halaqat.curriculum` (أو `kind`).
- **D2 ✅ — الارتباط:** حلقة المدرّس بلا أمير تُربط بـ**المربع المشرف** (`venue.orgUnitId` = مسار المربع) فيراها المشرف ضمن نطاقه.
- **D3 ✅ — المالية:** **«على بصيرة» مالية** (ساعات × سعر)؛ **التحفيظ والرشيدي غير ماليّين** (متابعة تربوية فقط). المالية تتبع المنهج: `curriculum === 'baseera'` فقط يُغذّي المالية.
- **D4 ✅ — التحفيظ القديم:** تبويب «التحفيظ» داخل المسجد (المرتبط بأمير) **يبقى كما هو** للمساجد؛ وتحفيظ المدرّس بلا مسجد يُمثَّل عبر النموذج الجديد (`curriculum='tahfeez'`). توحيدٌ كامل لاحقاً إن لزم (الأكثر منطقية: لا كسر للقائم).

## 13) النموذج النهائي المعتمد (curriculum)
- `halaqat.curriculum TEXT DEFAULT 'baseera'` — القيم: `baseera | tahfeez | rashidi | general`.
- المالية: تُحتسَب ساعات الدروس فقط حين `curriculum === 'baseera'`.
- المدرّس يملك حلقات بأيّ منهج؛ تقرير المدرّس يجمع: عدد الطلاب + الدروس (ما درّسه) + الساعات + التقييم/المتابعة — موحّداً عبر المناهج.
- ثوابت المناهج في ملف مشترك (مثل `lib/curricula.ts`) للتسميات العربية.
